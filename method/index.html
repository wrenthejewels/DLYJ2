How the model works

As AI gets better, which parts of your role come under pressure first, which parts still carry bargaining power, and what does that imply for the role as a whole? This page explains the live model in both plain language and formal terms.

In plain English

The live model works in six steps.

  1. It anchors you to a mapped occupation so the model starts from a real role rather than a vague title.
  2. It builds that role from the occupation task inventory, reviewed task expansions, and task-family structure.
  3. It reweights the role using your selected tasks so the model reflects your version of the occupation rather than the occupation average.
  4. It converts your role-refinement answers into a structured profile about function retention, sign-off, context load, adoption readiness, augmentation fit, and substitution pressure.
  5. It scores direct AI pressure, dependency spillover, retained leverage, and wave timing across the task graph.
  6. It turns those structural signals into a role-fate label, a wave trajectory, and a separate recomposition read.

Under the hood, the page still runs on a formal scoring model. The live input tuple is roughly:

Input tuple
$$x = (\text{roleCategory},\ \text{occupationId},\ \text{answers},\ \text{questionnaireProfile},\ \text{seniorityLevel},\ \text{dominantTaskIds},\ \text{criticalTaskIds},\ \text{aiSupportTaskIds},\ \text{supportTaskIds})$$

The task ids come from the five structured selectors on the main page: primary task, secondary task, value-defining task, AI-assisted task, and support/spillover task. The questionnaire answers are also converted into a structured role-refinement profile before scoring.

What the model looks at

The model does not guess from your title alone. It combines several layers of data:

  • A mapped occupation baseline from the launch occupation set
  • An occupation task inventory built from O*NET tasks plus reviewed role-graph expansions
  • Task dependency edges, so support work can lose value when exposed core work changes
  • Task-level evidence, with direct evidence where available and cluster-level fallback where it is not
  • Occupation priors, adaptation priors, and labor-market context such as projected growth and unemployment series
  • Your task selections, hierarchy level, and role-refinement profile

So the page is not asking only "can AI do this task?" It is also asking "if AI gets good at this work, what happens to the rest of the role around it, and does a coherent human role still remain?"

How it builds your role

The model starts from the selected occupation's task inventory and a task-family blend. That gives it a default picture of what the role is usually made of before personalization.

Baseline blend
$$\tilde s_i = 0.65\, s_i^{\text{cluster}} + 0.35\, s_i^{\text{inventory}}, \qquad s_i^{(0)} = \frac{\tilde s_i}{\sum_j \tilde s_j}$$

That baseline is then tilted by your selected task mix. Primary and value-defining work receive the strongest uplift, because they matter most for what the role actually is.

Override mass
$$o_i = w_i^{\text{explicit}} + 0.16\mathbf{1}[i\in \text{dominant}] + 0.10\mathbf{1}[i\in \text{critical}]$$
$$s_i = \frac{s_i^{(0)} + o_i}{\sum_j (s_j^{(0)} + o_j)}$$

In plain English, this is the model's answer to "what is your version of this occupation actually made of?"

How your answers matter

The role-refinement layer does not directly output the final verdict. It changes the conditions around the work: how observable the work is, how much of the function must still stay human, how much sign-off or liability remains, how bottlenecked the workflow is, and how quickly an organization could embed AI into it.

Normalization
$$q_k = \operatorname{clamp}\!\left(\frac{Q_k-1}{4},0,1\right), \qquad s = \operatorname{clamp}\!\left(\frac{\text{seniorityLevel}-1}{4},0,1\right)$$

The current UI still starts from question answers, but the engine turns those into a structured profile with fields such as function_centrality, human_signoff_requirement, liability_and_regulatory_burden, relationship_ownership, exception_and_context_load, workflow_decomposability, organizational_adoption_readiness, ai_observability_of_work, dependency_bottleneck_strength, augmentation_fit, and substitution_risk_modifier.

Composite signals
$$\text{functionRetention} = \operatorname{avg}(\text{functionCentrality},\ \text{signoff},\ \text{liability},\ \text{relationshipOwnership})$$
$$\text{capabilitySignal} = \operatorname{avg}(\text{aiObservability},\ \text{workflowDecomposability},\ \text{substitutionRiskModifier})$$
$$\text{couplingProtection} = \operatorname{avg}(\text{functionRetention},\ \text{exceptionLoad},\ \text{dependencyBottlenecks},\ \text{externalTrust})$$
$$\text{adoptionPressure} = \text{organizationalAdoptionReadiness}$$
Friction dimensions
$$f_i = (f^{\text{exception}}, f^{\text{accountability}}, f^{\text{judgment}}, f^{\text{document}}, f^{\text{tacit}})$$
$$f^{\text{exception}}=\text{exceptionAndContextLoad}, \quad f^{\text{accountability}}=\operatorname{avg}(\text{signoff},\ \text{liability},\ \text{functionCentrality})$$
$$f^{\text{judgment}}=\operatorname{avg}(\text{functionCentrality},\ 1-\text{executionVsJudgmentMix},\ \text{relationshipOwnership})$$
$$f^{\text{document}}=\operatorname{avg}(\text{aiObservability},\ \text{workflowDecomposability}), \quad f^{\text{tacit}}=\operatorname{avg}(\text{exceptionLoad},\ \text{dependencyBottlenecks},\ \text{externalTrust})$$

In plain English, the questionnaire tells the model whether AI mainly helps the role, compresses the role, or still leaves a human-retained core because of function, accountability, trust, and messy exceptions.

How the model decides which work is easier to absorb

Each task family gets an automation-difficulty score. This is not just "AI can do the task" versus "AI cannot." It is the model's estimate of how hard it is for organizations to actually absorb that work into workflows at scale.

Task-family friction
$$F_i = 0.25f^{\text{accountability}}_i + 0.22f^{\text{judgment}}_i + 0.22f^{\text{tacit}}_i + 0.18f^{\text{exception}}_i + 0.13(1-f^{\text{document}}_i)$$
Difficulty score
$$D_i = \operatorname{clamp}\!\left(0.40F_i + 0.25(0.35h_i) + 0.25R_i + 0.08\cdot\text{functionRetention} + 0.04\cdot\text{augmentationFit} + 0.08\cdot\text{substitutionRiskModifier},\,0.02,\,0.98\right)$$
$$D_i \leftarrow \operatorname{clamp}\!\left(D_i + 0.10B_i + 0.05C_i + 0.04\max(0,V_i-0.5) - 0.07A_i + 0.08K_i,\,0.02,\,0.98\right)$$

h_i is human advantage, R_i is empirical resistance, B_i is bargaining-power weight, C_i is core-task share, V_i is value centrality, A_i is AI-support observability, and K_i indicates a role-critical family. Lower difficulty means the task family is easier to absorb sooner.

After that, each task family is placed into an approximate timing bucket: current wave, next wave, or distant wave.

Wave assignment
$$\text{wave}_i = \begin{cases} \text{current}, & D_i \le 0.35 \\ \text{next}, & 0.35 < D_i \le 0.65 \\ \text{distant}, & D_i > 0.65 \end{cases}$$

How direct risk and spillover work

This is the part most users care about. The model distinguishes between two kinds of pressure:

  • Direct pressure: AI gets good at the task itself.
  • Indirect spillover: the task is not easy to automate, but it exists mainly to support work that is under pressure.

The task graph is how the model captures that second effect. If a support task mainly exists because another task exists, then automating the core task can still reduce the need for the support task.

Task share
$$s_t^{\text{raw}} = s_{c(t)} \cdot \frac{\text{timeSharePrior}_t}{\sum_{j \in c(t)} \text{timeSharePrior}_j}$$
$$m_t = 1 + 0.40\mathbf{1}[t\in \text{dominant}] + 0.12\mathbf{1}[t\in \text{critical}] + 0.18\mathbf{1}[t\in \text{support}]$$
$$s_t = \frac{s_t^{\text{raw}}m_t}{\sum_j s_j^{\text{raw}}m_j}$$
Task pressure
$$p_t^{\text{direct}} = \operatorname{clamp}\!\left(0.68(1-D_{c(t)}) + 0.20\alpha_{c(t)} + 0.12A_t - 0.14\mathbf{1}[t\in \text{aiSupport}] + 0.03\mathbf{1}[t\in \text{support}],\,0,\,1\right)$$
$$p_t^{\text{spill}} = \sum_{(t,u)\in \mathcal{D}} d_{tu}\, p_u^{\text{direct}} \max(V_u,B_u) + 0.12\mathbf{1}[t\in \text{support}]$$
Retained leverage
$$L_t = \operatorname{clamp}\!\left(B_t(1-p_t^{\text{direct}})(1-p_t^{\text{spill}}) + 0.10\mathbf{1}[t\in \text{core}] + 0.08\mathbf{1}[t\in \text{critical}] + 0.08\mathbf{1}[t\in \text{aiSupport}] + 0.60\,\text{elevationBoost}_t,\,0,\,1\right)$$

The task graph also produces role-level summary scores:

  • direct_exposure_pressure
  • indirect_dependency_pressure
  • retained_leverage_score
  • residual_role_integrity
  • exposed_core_share and retained_core_share
  • direct_coverage_ratio, which shows how much of the task graph has direct task evidence rather than fallback estimates

Those task-graph signals do most of the real explanatory work on the page.

How the final fate label is chosen

Once the model knows what work is exposed, what work spills over, and what work still carries leverage, it maps the role into one of seven organizational outcomes:

  • Augmented — AI speeds up the work, but the role still clearly exists.
  • Compressed — the role still exists, but firms may need fewer people doing it.
  • Elevated — the execution layer shrinks and the retained role becomes more judgment-heavy.
  • Split — lower-cost execution separates from a smaller higher-judgment core.
  • Expanded — AI increases output or span of control enough that demand grows.
  • Collapsed — the role's core reason to exist weakens sharply.
  • Mixed transition — no single path dominates.

The live classifier is a rule-based tree rather than a trained neural model. It mainly looks at direct exposure pressure, dependency spillover, retained leverage, residual role integrity, exposed versus retained core share, next-wave retained share and coherence, and a simple demand modifier from BLS growth.

Demand modifier
$$M_{\text{demand}} = \operatorname{clamp}\!\left(\frac{\text{projectionGrowthPct}+2}{10},\,0,\,1\right)$$

High retained leverage, strong next-wave coherence, and low direct pressure push roles toward Augmented or Elevated. High demand growth can push some roles into Expanded. Large exposed core share and weak retained integrity pull the model toward Compressed, Split, or in the most severe cases Collapsed. When the signals conflict, the model falls back to Mixed transition.

Recomposition layer

The page also shows a secondary recomposition panel. This is not the same as the final fate label. It is a narrower estimate of how much exposed work looks technically compressible and how much of that compression may convert into fewer labor hours in firms.

Compression and conversion
$$W = \operatorname{clamp}\!\left(\text{currentWaveAbsorbed}(1-0.20\cdot\text{couplingProtection})(1-0.10\cdot\text{functionRetention}),\,0,\,1\right)$$
$$O = \operatorname{clamp}\!\left(0.30\text{adoptionPressure} + 0.25(1-\text{couplingProtection}) + 0.20\text{currentWaveAbsorbed} + 0.10(1-f^{\text{accountability}}) + 0.08(1-f^{\text{judgment}}) + 0.07f^{\text{document}} - 0.05\text{signoff} - 0.05\text{liability},\,0,\,1\right)$$
$$\text{substitutionPotential} = W \cdot O, \qquad \text{substitutionGap} = W - \text{substitutionPotential}$$

Those raw values are then nudged by the task graph. More direct exposure pushes compression higher, while stronger retained leverage and dependency bottlenecks resist full organizational conversion. That is why the recomposition panel can disagree with the final fate label: a role may have technically compressible work and still retain a strong human core.

Why the page looks the way it does

The visible role fate map is built from the underlying task table. Each column answers a different user question:

  • Current role: what takes up the role today?
  • Bargaining power: what most explains why this role exists?
  • Direct pressure: what AI pressures first?
  • Indirect spillover: what shrinks because it depends on exposed work?
  • Retained leverage: what still justifies the role after AI gets better?

Formally, the page ranks tasks by different task-level signals:

Client rankings
$$\text{Current role} = \operatorname{TopK}_t(s_t)$$
$$\text{Bargaining power} = \operatorname{TopK}_t\left(s_t \cdot \max(B_t,V_t)\right)$$
$$\text{Direct pressure} = \operatorname{TopK}_t(s_t \cdot p_t^{\text{direct}})$$
$$\text{Indirect spillover} = \operatorname{TopK}_t(s_t \cdot p_t^{\text{spill}})$$
$$\text{Retained leverage} = \operatorname{TopK}_t(\text{retainedShare}_t \cdot L_t)$$

What this model does not do

This model is useful, but it is not magic. It emits descriptive confidence and coverage fields, not a full probability distribution over the future.

  • Role-fate confidence is a margin-style score from the current rule set, not a calibrated probability.
  • Task evidence coverage still varies by occupation and task family.
  • User task input is still selector-based rather than weighted by rough percentage buckets.
  • The classifier is calibrated heuristically against anchor occupations, not trained on a labeled dataset.
  • The wave system still exists internally, so role fate currently sits on top of the wave engine rather than fully replacing it.
