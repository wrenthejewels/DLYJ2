How the model works

As AI gets better, which parts of your role come under pressure first, which parts still protect the role, and what does that imply for the job as a whole? This page explains the live model in user-facing terms first, then more formally.

In plain English

The live model works in six steps.

  1. It anchors you to a mapped occupation so the model starts from a real role rather than a vague title.
  2. It builds that role from the occupation task inventory, reviewed task expansions, and reviewed function anchors.
  3. It lets you edit that role composition by removing or adding source-bucketed tasks, adjusting which functions count, and optionally declaring a few support links between selected tasks or extra task-to-function links.
  4. It converts your role-refinement answers into a structured profile about function retention, sign-off, context load, adoption readiness, augmentation fit, and substitution pressure.
  5. It scores direct AI pressure, dependency spillover, retained leverage, and wave timing across the task graph.
  6. It turns those structural signals into a role-fate label, a wave trajectory, and a separate recomposition read.

Under the hood, the page still runs on a formal scoring model. The live input tuple is roughly:

Input tuple
$$x = (\text{roleCategory},\ \text{occupationId},\ \text{questionnaireProfile},\ \text{seniorityLevel},\ \text{compositionEdits},\ \text{dependencyEdits})$$

The simplest mental model is this: tasks describe how the job gets done, while functions describe what the human role is ultimately there to own. AI often pressures the how before it eliminates the why. The model exists to track that transition rather than assuming that exposed tasks automatically mean the whole role disappears.

The editable task ids now come from the role breakdown on the main page: O*NET tasks, reviewed public-posting tasks, reviewed role-review tasks, and value-defining functions. The editor also shows which functions the selected tasks mainly support, and it accepts optional user-declared support links when one selected task mainly exists to enable another or when a selected task should feed an additional selected function. The role-refinement panel writes a native structured profile before scoring, so the runtime is no longer relying on old numbered question ids in the live app path.

What the model looks at

The model does not guess from your title alone. It combines several layers of data from external research and primary observation:

Primary task evidence

  • Anthropic Economic Index (2026-01-15 release) — the main live-task empirical source feeding the stack. Normalized from Claude.ai and first-party API task telemetry with O*NET task classification, collaboration patterns, human-only ability flags, task success rates, and human/AI time estimates. In the current browser scorer, these rows feed the top-priority live_task_evidence tier inside the task-source resolver, preserve Anthropic-specific observation metadata like usage share, and also feed the reviewed cluster priors that still seed the baseline difficulty model. The checked-in normalized layer currently contains 530 task evidence rows across all 34 modeled occupations.

Structural backbone

  • O*NET Database (v30.1) — occupation taxonomy, task statements, work context, skills, abilities, and job zones. Provides the task inventory that the role is built from.
  • Reviewed task and role-graph expansions — internal task-gap expansions curated from job-description review for occupations underdescribed by O*NET alone, plus task dependency edges and role-function anchors.
  • Reviewed function-accountability priors — manually authored role-family defaults and occupation-specific overrides describing judgment, trust, liability, authority, and bargaining retention at the function level. These are used to build the function graph and the guardrail signals that sit above the task graph.

Labor-market context

  • BLS Occupational Employment and Wage Statistics (May 2024) — employment counts and median wages for occupation context.
  • BLS Occupational Projections (2024–2034) — projected employment growth, used in the demand modifier for fate classification.
  • BLS Current Population Survey — monthly unemployment series by occupation group, used for the labor-market context chart.
  • BLS Occupational Requirements Survey (2025 preliminary with 2023 backstop) — calibration-only structural context for human-retention checks. In the current repo it does not directly change runtime task scores; it is used to pressure-test the model’s accountability and autonomy guardrails against observed work requirements.

Benchmark validation layers

Most of these academic benchmarks are used to validate directional rankings rather than to drive the public runtime directly. The main exception is task-level GPT benchmark labels, which now participate as a supporting task-evidence tier in the live task-source resolver. Each source is normalized to a common occupation or task key and compared against the live exposure stack:

  • AIOE (AI Occupational Exposure, 2023) — occupation-level AI exposure measure including language-modeling-specific and ability-level exposure values.
  • Webb AI Exposure (2020, NBER WP 24762) — patent-based occupation-level AI, robotics, and software exposure measures.
  • Brynjolfsson, Mitchell & Rock (SML, AEA P&P 2018) — suitability for machine learning at the occupation level.
  • GPTs are GPTs (Eloundou et al., 2023) — GPT-4 and human annotations for task-level and occupation-level automation ratings, plus autoScores for Webb, SML, robotics, and software exposure.

User inputs

  • Your task selections, hierarchy level, optional support-link declarations, optional task-to-function link edits, and role-refinement profile.

So the page is not asking only "can AI do this task?" It is also asking "if AI gets good at this work, what happens to the rest of the role around it, and does a coherent human role still remain?"

How task evidence becomes exposure scores

The data sources above do not all carry equal weight. The live browser scorer is currently hybrid. It still starts from reviewed task-family priors for the fallback difficulty layer and shrinks those priors toward an occupation-level exposure prior. But that is no longer the whole baseline path: when a task family has enough strong resolved task-level evidence, the runtime first shifts that cluster baseline toward a task-first evidence estimate before projecting it onto the active task graph. Then, when an individual task itself has high enough direct reliability, that task can promote again into its own task-first baseline rather than only inheriting the cluster seed. A runtime task-source resolver chooses among live Anthropic task evidence, reviewed task estimates, benchmark task labels, and only then proxy fallback. Any remaining resolved task-level evidence can still blend into both a task’s final automation difficulty and its final direct-pressure score. After the task table is scored, the public wave engine is recomputed from the task-derived cluster bundle rather than preserved from the older pre-task cluster bundle.

Evidence layers

  1. Task-level evidence tiers — the runtime first resolves the best available task-level evidence from task_source_evidence.csv. The live precedence is live_task_evidence, then reviewed_task_estimate, then benchmark_task_label. Those tiers can all affect the live task score when reliability is high enough, and the strongest task rows can now become their own baseline source rather than only adjusting a cluster baseline after the fact.
  2. Cluster-level priors — task-family priors carrying partial_automation_likelihood and high_automation_likelihood estimates with an evidence confidence score and primary source tags. This is still the fallback live scoring layer for automation difficulty when cluster-level task evidence is too thin to support the task-first baseline path.
  3. Occupation-level priors — broader occupation exposure priors carrying an automation_score. A separate occupation adaptation prior feeds personalization and function-layer demand signals later in the run. The occupation exposure prior is the broadest fallback when cluster evidence is weak.

The broader pipeline still keeps many comparison layers separate, but task_source_evidence.csv is now a real runtime input rather than a comparison-only artifact. The live browser scorer now has both a coverage-aware task-first cluster-baseline path and a task-first task-baseline path layered on top of the older cluster-prior stack. That task-first task path is source-aware and mapping-confidence-aware, so stronger live or reviewed task evidence can promote more readily than weaker benchmark-only evidence. Low-coverage tasks still inherit a cluster-seeded fallback, so this is still a hybrid model rather than a universal task-first prior system.

Shrinkage toward prior

The core shrinkage step in the live scorer still begins at the cluster level. When a task family has strong prior evidence but weak task-level coverage, its own automation estimate dominates. When task-level coverage and reliability are strong enough, the runtime first shifts that cluster baseline toward the resolved task-evidence estimate. Then, for especially strong task rows, the task-level baseline itself can shift again toward the resolved task evidence. When evidence is sparse, the cluster estimate is pulled toward the broader occupation prior:

Shrinkage function
$$\hat\theta = \lambda\,\theta_{\text{observed}} + (1-\lambda)\,\theta_{\text{prior}}$$

where λ is the evidence reliability score. High reliability means the observation stands on its own; low reliability means it defers to the prior.

Evidence reliability

The model calculates reliability differently depending on the evidence layer. In the current browser scorer, cluster-prior reliability directly affects the fallback baseline difficulty estimate, while task-level evidence reliability both feeds diagnostics and gates whether resolved task evidence is allowed to alter cluster baselines, task baselines, task difficulty, and task pressure. The task-first task-baseline gate is also source-aware and damped by task mapping confidence.

Task-level evidence reliability now depends on source role. For live Anthropic task evidence, the runtime weights stated confidence by observation volume. Internal stub sources (placeholder rows with no real observation) receive zero reliability:

Task evidence reliability
$$\lambda_{\text{task}} = \operatorname{clamp}\!\left(\text{confidence} \cdot \frac{n_{\text{tasks}}}{n_{\text{tasks}} + 40},\,0.05,\,0.98\right)$$

So a task with 40 observations gets roughly half-weight from its own evidence; a task with 400 observations gets roughly 91% weight. For reviewed task estimates, benchmark task labels, and proxy rows, the runtime instead uses stated confidence, normalized evidence weight, and source-role / promotion multipliers. In the current runtime this resolved reliability informs evidence confidence and coverage reporting, can promote high-reliability tasks into a task-first baseline with stricter promotion for benchmark labels than for live or reviewed task evidence, and once it clears the runtime threshold of 0.20 it can also blend resolved task evidence into the task’s final automation-difficulty and direct-pressure scores.

Task evidence blend gate
$$w_t^{\text{evidence}} = \begin{cases} 0, & \lambda_{\text{task}} \le 0.20 \\ \operatorname{clamp}\!\left(0.85\cdot\frac{\lambda_{\text{task}}-0.20}{1-0.20},\,0,\,0.85\right), & \lambda_{\text{task}} > 0.20 \end{cases}$$

If a task already clears the stronger task-first baseline gate, the runtime reduces the remaining blend weight by the amount already consumed in that baseline promotion instead of applying the same evidence twice.

Task ease signal
$$q_t^{\text{ease}} = 0.65\,\text{automationScore}_t + 0.25\,\text{exposureScore}_t + 0.10\,\text{augmentationScore}_t$$
Task evidence signal
$$q_t^{\text{direct}} = 0.50\,\text{automationScore}_t + 0.35\,\text{exposureScore}_t + 0.15\,\text{augmentationScore}_t$$
Resolved task evidence
$$\omega_{t,s} = \lambda_{t,s}\cdot \text{evidenceWeight}_{t,s}\cdot m(\text{sourceRole}_s)$$
$$\tilde a_t = \frac{\sum_{s \in \mathcal{S}^{\text{task}}_t} \omega_{t,s}\,a_{t,s}}{\sum_{s \in \mathcal{S}^{\text{task}}_t} \omega_{t,s}}, \qquad \tilde e_t = \frac{\sum_{s \in \mathcal{S}^{\text{task}}_t} \omega_{t,s}\,e_{t,s}}{\sum_{s \in \mathcal{S}^{\text{task}}_t} \omega_{t,s}}$$

Here the resolver only averages over the promoted task-level tiers for the task: live Anthropic evidence first, then reviewed task estimates, then benchmark task labels. Cluster proxies and fallback proxies remain visible as fallback metadata, but they do not become part of the task-evidence blend itself.

Cluster prior reliability starts from stated evidence confidence but penalizes sources that rely on internal stubs rather than real observations:

Prior reliability
$$\lambda_{\text{prior}} = \operatorname{clamp}\!\left(\text{confidence} \cdot \rho_{\text{source}},\,0.05,\,0.98\right)$$
$$\rho_{\text{source}} = \begin{cases} 0.55, & \text{stub-only sources} \\ 0.78, & \text{stub + Anthropic mix} \\ 1.0, & \text{no stub sources} \end{cases}$$

From evidence to empirical resistance

The live difficulty engine produces the empirical resistance term Ri from the cluster prior layer. For each task family, the model computes an empirical ease score by shrinking the cluster's automation likelihood toward the occupation-level automation prior, then inverts it:

Empirical resistance
$$e_i = \operatorname{shrink}\!\left(\operatorname{avg}(p_i^{\text{partial}},\, p_i^{\text{high}}),\;\theta_{\text{occ}}^{\text{auto}},\;\lambda_{\text{prior}},\;0.25\right)$$
$$R_i = 1 - e_i$$

where pipartial and pihigh are the cluster’s partial and high automation likelihoods, θoccauto is the occupation-level automation score, and λprior is the cluster’s evidence reliability. When cluster evidence is strong, Ri reflects the cluster’s own resistance. When evidence is weak, it falls back toward the occupation average.

Human advantage

Each task family also carries a static human advantage score hi that reflects how much the work type inherently depends on human presence, relationships, or embodied judgment. These are set per cluster type, ranging from 1.0 for client interaction and relationship management down to 0.05 for routine execution. The engine first scales this to 0.35hi, then applies the human-advantage weight inside the difficulty formula. Human advantage also interacts with function retention in the task-graph adjustment step.

Cluster friction profiles

Each task family has a baseline friction profile across five dimensions: exception burden, accountability load, judgment requirement, document intensity, and tacit context dependence. These baselines are then adjusted by the user’s questionnaire responses using a delta-from-neutral blend:

Friction blending
$$f_i^{d} = \operatorname{clamp}\!\left(f_{\text{profile}}^{d} + (f_{\text{user}}^{d} - 0.5) \cdot 0.30,\,0,\,1\right)$$

This means the cluster’s structural friction profile is the anchor, and the user’s answers can shift each dimension by up to ±0.15 from the baseline. An answer at the midpoint changes nothing; extremes pull the friction up or down.

Task-graph signals

The task inventory also contributes four signals per task family: bargaining power weight Bi (how much the task explains why the role exists), core share Ci (proportion of core-criticality tasks), value centrality Vi (how central the task is to the role’s value delivery), and AI support observability Ai (how visible and measurable the task’s output is to AI systems). These are averaged from the individual tasks within each family and feed into the difficulty and pressure formulas documented in subsequent sections.

Evidence confidence at the task level

Each individual task in the role breakdown carries a composite evidence confidence that averages four signals: the cluster’s evidence confidence, the task’s source confidence from the inventory, the direct evidence confidence (if any), and the task-to-cluster mapping confidence. The model also tracks a binary has_direct_evidence flag and a direct_coverage_ratio at the role level, so the UI can communicate how much of the role resolved to task-level evidence rather than proxy fallback. In the current engine, that same task-level reliability also decides whether the task gets a live difficulty and direct-pressure blend.

How it builds your role

The model starts from the selected occupation's task inventory and a task-family blend. It then links those tasks into one or more reviewed role functions, so the baseline is not just a flat task list. That gives it a default picture of what the role is usually made of and what the role is trying to accomplish before personalization.

That distinction matters. Tasks are the operating layer: drafting, reviewing, coordinating, documenting, analyzing, following up. Functions are the purpose layer: client ownership, decision support, compliance judgment, cross-functional orchestration, accountable delivery. A user could try to reason only at the function level, but functions are too broad to show where pressure starts. The model uses tasks to locate the pressure, then functions to decide whether the human-owned core still survives after those tasks change.

Baseline blend
$$\tilde s_i = 0.65\, s_i^{\text{cluster}} + 0.35\, s_i^{\text{inventory}}, \qquad s_i^{(0)} = \frac{\tilde s_i}{\sum_j \tilde s_j}$$

That baseline is then filtered through your composition edits. Removing tasks takes them out of the active run, adding reviewed tasks puts them into the active run, and function edits renormalize the role's retained-function mix. The live scorer now uses the reviewed baseline task-to-function graph directly rather than showing it only for reference. Optional support links can add spillover paths that do not exist in the default graph, and optional task-to-function links act as overrides or additions when the default graph misses an important connection.

Edited composition
$$\mathcal{T}^{*} = (\mathcal{T}^{(0)} \setminus \mathcal{T}^{-}) \cup \mathcal{T}^{+}, \qquad \mathcal{F}^{*} = (\mathcal{F}^{(0)} \setminus \mathcal{F}^{-}) \cup \mathcal{F}^{+}$$
$$s_i = \frac{\mathbf{1}[i\in \mathcal{T}^{*}]\, \tilde s_i}{\sum_{j \in \mathcal{T}^{*}} \tilde s_j}, \qquad w_f = \frac{\mathbf{1}[f\in \mathcal{F}^{*}]\, \tilde w_f}{\sum_{g \in \mathcal{F}^{*}} \tilde w_g}$$

In plain English, this is the model's answer to "what is your version of this occupation actually made of?" The extra function layer matters because two roles can have similarly exposed tasks but still differ sharply in whether the human role survives as the accountable core.

Users can change this structure directly. Adding or removing tasks changes what work the role contains. Editing task share changes what matters most. Adding support links changes how pressure can spread through the role. Adding a task-to-function link tells the model that a task contributes to a more central function, which can increase that task's effective centrality, bargaining weight, and sometimes its treatment as core work. But even without custom edits, the default reviewed task-to-function bindings now already affect live scoring.

How your answers matter

The role-refinement layer does not directly output the final verdict. It changes the conditions around the work: how observable the work is, how much of the function must still stay human, how much sign-off or liability remains, how bottlenecked the workflow is, and how quickly an organization could actually embed AI into it.

Normalization
$$r_k = \operatorname{clamp}\!\left(\frac{R_k-1}{4},0,1\right), \qquad s = \operatorname{clamp}\!\left(\frac{\text{seniorityLevel}-1}{4},0,1\right)$$

The hierarchy input mainly works through that s term. Higher hierarchy levels tell the model that your version of the occupation is more likely to carry sign-off, coordination, stakeholder alignment, and retained human accountability. Lower hierarchy levels push the same occupation toward more execution-heavy, more decomposable work. It does not switch you into a different occupation; it changes how the model interprets your version of the selected occupation.

The live UI now captures named role-refinement responses directly, then turns them into a structured profile with fields such as function_centrality, human_signoff_requirement, liability_and_regulatory_burden, relationship_ownership, exception_and_context_load, workflow_decomposability, organizational_adoption_readiness, ai_observability_of_work, dependency_bottleneck_strength, augmentation_fit, and substitution_risk_modifier.

The visible prompts are still simple, but under the hood the important distinction is this: the questionnaire is trying to measure what part of the role stays human-retained even if AI helps with more of the underlying work.

Composite signals
$$\text{functionRetention} = \operatorname{avg}(\text{functionCentrality},\ \text{signoff},\ \text{liability},\ \text{relationshipOwnership})$$
$$\text{capabilitySignal} = \operatorname{avg}(\text{aiObservability},\ \text{workflowDecomposability},\ \text{substitutionRiskModifier})$$
$$\text{couplingProtection} = \operatorname{avg}(\text{functionRetention},\ \text{exceptionLoad},\ \text{dependencyBottlenecks},\ \text{externalTrust})$$
$$\text{adoptionPressure} = \text{organizationalAdoptionReadiness}$$
Friction dimensions
$$f_i = (f^{\text{exception}}, f^{\text{accountability}}, f^{\text{judgment}}, f^{\text{document}}, f^{\text{tacit}})$$
$$f^{\text{exception}}=\text{exceptionAndContextLoad}, \quad f^{\text{accountability}}=\operatorname{avg}(\text{signoff},\ \text{liability},\ \text{functionCentrality})$$
$$f^{\text{judgment}}=\operatorname{avg}(\text{functionCentrality},\ 1-\text{executionVsJudgmentMix},\ \text{relationshipOwnership})$$
$$f^{\text{document}}=\operatorname{avg}(\text{aiObservability},\ \text{workflowDecomposability}), \quad f^{\text{tacit}}=\operatorname{avg}(\text{exceptionLoad},\ \text{dependencyBottlenecks},\ \text{externalTrust})$$

In plain English, the questionnaire tells the model whether AI mainly helps the role, compresses the role, or still leaves a human-retained core because of function, accountability, trust, and messy exceptions.

When the selected occupation has a reviewed function graph, the model blends function-level accountability signals into the questionnaire-derived values before scoring. This anchors the signals to the occupation's actual authority and delegation structure rather than relying on user responses alone.

Those function-level signals are not inferred directly from Anthropic telemetry. They are reviewed function priors first, then summarized mathematically. Each occupation gets one or more function anchors with weighted values for judgment, trust, regulatory burden, human authority, and bargaining retention. The live engine collapses those anchors into weighted averages, blends them into the runtime signals below, and then uses the active task-to-function graph to compute a second live readout of function exposure and retained function strength.

Function-layer blending
$$\text{functionRetention} \leftarrow 0.72\,\text{functionRetention} + 0.14\,\text{humanAuthorityReq} + 0.14\,\text{bargainingPowerRetention}$$
$$\text{couplingProtection} \leftarrow 0.78\,\text{couplingProtection} + 0.10\,\text{delegabilityGuardrail} + 0.06\,\text{trustReq} + 0.06\,\text{judgmentReq}$$
$$\text{augmentationFit} \leftarrow 0.80\,\text{augmentationFit} + 0.10\,\text{bargainingPowerRetention} + 0.10\,\text{judgmentReq}$$
$$\text{substitutionRiskModifier} \leftarrow 0.82\,\text{substitutionRiskModifier} + 0.10(1-\text{delegabilityGuardrail}) + 0.08(1-\text{humanAuthorityReq})$$

The same blending applies to three friction dimensions: accountability_load, judgment_requirement, and tacit_context_dependence each blend at 0.75 questionnaire / 0.25 function-graph. Occupations without a reviewed function graph skip this step entirely and use the questionnaire signals as-is.

How the model decides which work is easier to absorb

Each task family gets an automation-difficulty score. This is not just "AI can do the task" versus "AI cannot." It is the model's estimate of how hard it is for organizations to actually absorb that work into workflows at scale, after accounting for retained function, cluster-level evidence strength, and workflow friction.

Task-family friction
$$F_i = 0.25f^{\text{accountability}}_i + 0.22f^{\text{judgment}}_i + 0.22f^{\text{tacit}}_i + 0.18f^{\text{exception}}_i + 0.13(1-f^{\text{document}}_i)$$
Difficulty score — base
$$D_i = \operatorname{clamp}\!\left(0.40F_i + 0.25(0.35\,h_i) + 0.25R_i + 0.10\cdot\text{couplingProtection},\,0.02,\,0.98\right)$$
Difficulty score — task-graph adjustments
$$D_i \leftarrow \operatorname{clamp}\!\left(D_i + 0.08\,h_i\cdot\text{functionRetention} + 0.04\,\text{augmentationFit} + 0.10B_i + 0.05C_i + 0.04\max(0,V_i-0.5) - 0.07A_i - 0.08\,\text{substitutionRiskModifier},\,0.02,\,0.98\right)$$
Difficulty score — criticality boost
$$D_i \leftarrow \operatorname{clamp}\!\left(D_i + 0.08,\,0.02,\,0.98\right) \quad \text{if task family is role-critical}$$

h_i is human advantage, R_i is empirical resistance, B_i is bargaining-power weight, C_i is core-task share, V_i is value centrality, A_i is AI-support observability. The base step combines friction, human advantage, empirical resistance, and coupling protection. The second step adjusts for task-graph signals: tasks with high bargaining power or value centrality resist automation, while high AI-support observability and substitution risk lower difficulty. The criticality boost applies only to task families flagged as role-critical. Lower difficulty means the task family is easier to absorb sooner.

That cluster-family difficulty is still the baseline. The live scorer then projects it onto each active task row and lets reliable resolved task evidence alter the task’s own difficulty before wave timing is recomputed at the cluster-summary level.

Task difficulty blend
$$D_t = \operatorname{clamp}\!\left((1-w_t^{\text{evidence}})\,D_{c(t)} + w_t^{\text{evidence}}(1-q_t^{\text{ease}}),\,0.02,\,0.98\right)$$

After that, each task row is placed into an approximate timing bucket: current wave, next wave, or distant wave. The public cluster wave timing is then recomputed from the task-derived cluster summaries rather than copied from the original cluster bundle.

Wave assignment
$$\text{wave}_i = \begin{cases} \text{current}, & D_i \le 0.35 \\ \text{next}, & 0.35 < D_i \le 0.65 \\ \text{distant}, & D_i > 0.65 \end{cases}$$

Within each wave, the model also estimates an absorption rate: how much of the task family's share is likely to be absorbed into AI-assisted workflows. This is not a binary on/off — it captures the realistic pace of organizational uptake.

Adoption realization
$$\text{adoptionRealization} = 0.92 + 0.16\cdot\text{adoptionPressure}$$
Absorption rate
$$a_i = \operatorname{clamp}\!\left(\text{adoptionRealization}\,(1 - 0.3D_i)\,(0.92 + 0.10A_i - 0.06C_i)\,(1 - 0.10\,\text{dependencyBottlenecks})\,(1 - 0.08\,\text{signoff}),\,0.45,\,0.95\right)$$

The task rows use the same basic absorption logic after the task-level difficulty blend, and the cluster-level wave engine then recomputes retained share and elevation from the task-derived cluster bundle. The absorbed share in the current wave feeds directly into the recomposition layer's workflow compression estimate.

How direct risk and spillover work

This is the part most users care about. The model distinguishes between two kinds of pressure:

  • Direct pressure: AI gets good at the task itself.
  • Indirect spillover: the task is not easy to automate, but it exists mainly to support work that is under pressure.

The task graph is how the model captures that second effect. If a support task mainly exists because another task exists, then automating the core task can still reduce the need for the support task.

Task share
$$s_t^{\text{raw}} = s_{c(t)} \cdot \frac{\text{timeSharePrior}_t}{\sum_{j \in c(t)} \text{timeSharePrior}_j}$$
$$s_t = \frac{\mathbf{1}[t\in \mathcal{T}^{*}]\, s_t^{\text{raw}}}{\sum_{j \in \mathcal{T}^{*}} s_j^{\text{raw}}}$$

If you manually edit a task's share in the role graph, the model substitutes your edited share before this normalization step. The live UI does not currently expose separate dominant-task, critical-task, or AI-support toggles, so the browser path does not apply extra task multipliers beyond the edited share itself.

Task pressure
$$p_{t,\text{base}}^{\text{direct}} = \operatorname{clamp}\!\left(0.68(1-D_t) + 0.20\alpha_{c(t)} + 0.12A_t,\,0,\,1\right)$$
$$p_t^{\text{direct}} = (1-w_t^{\text{evidence}})\,p_{t,\text{base}}^{\text{direct}} + w_t^{\text{evidence}}\,q_t^{\text{direct}}$$
$$p_t^{\text{spill}} = \sum_{(t,u)\in \mathcal{D}^{*}} d_{tu}\, p_u^{\text{direct}} \max(V_u,B_u)$$

The flow is: current task mix → direct pressure on exposed tasks → spillover into support tasks → surviving leverage among the tasks still tied to core functions. That is why the model is more useful than a simple "can AI do this function?" thought exercise. Looking only at functions is too coarse; looking only at tasks overstates replacement. The model uses both layers to ask whether a coherent human-owned role remains after pressure moves through the task map.

Retained leverage
$$L_t = \operatorname{clamp}\!\left(B_t(1-p_t^{\text{direct}})(1-p_t^{\text{spill}}) + 0.10\mathbf{1}[t\in \text{core}] + 0.60\,\text{elevationBoost}_t,\,0,\,1\right)$$

After the task table is scored, the live app now aggregates those task rows back into cluster summaries for the public cluster layer and recomputes the wave engine from that task-derived bundle. So the visible top exposed cluster, exposed-cluster list, retained-cluster list, elevated-cluster list, and wave cards are now task-derived rather than untouched copies of the pre-task cluster bundle.

Task-derived cluster summary
$$\bar p_i^{\text{direct}} = \frac{\sum_{t \in i} s_t\,p_t^{\text{direct}}}{\sum_{t \in i} s_t}, \qquad \bar p_i^{\text{spill}} = \frac{\sum_{t \in i} s_t\,p_t^{\text{spill}}}{\sum_{t \in i} s_t}$$
$$\text{exposedShare}_i = \sum_{t \in i} \text{exposedShare}_t, \qquad \text{retainedShare}_i = \sum_{t \in i} \text{retainedShare}_t$$

The task graph also produces role-level summary scores:

  • direct_exposure_pressure
  • indirect_dependency_pressure
  • retained_leverage_score
  • residual_role_integrity
  • exposed_core_share and retained_core_share
  • direct_coverage_ratio, which shows how much of the task graph resolved to task-level evidence rather than proxy fallback

Those task-graph signals do most of the real explanatory work on the page. The explanation panel is now generated from the current live run, so when you edit tasks, functions, support links, or task-to-function links, the explanation updates with the same pressure and counterweight logic used by the scorer. The composition editor exposes part of that logic earlier by showing the top function links for each selected task and by letting you add a few support links or task-to-function links when the default graph misses an obvious connection.

How the final fate label is chosen

Once the model knows what work is exposed, what work spills over, and what work still carries leverage, it maps the role into one of seven organizational outcomes:

  • Growing — AI creates more demand for this role than it displaces.
  • Reinforced — AI slots in as a productivity layer, not a structural threat. The role stays intact.
  • Narrowed upward — the execution layer shrinks and the retained role becomes more senior and judgment-heavy.
  • Polarized — the role fractures into a cheaper execution tier and a smaller, higher-judgment oversight tier.
  • Shrinking — the work stays, but fewer people are needed to cover it.
  • Hollowed out — AI reaches the core of what justifies this role existing as a distinct position.
  • Unclear trajectory — competing pressures with no dominant path yet.

The live classifier is a rule-based tree rather than a trained neural model. It mainly looks at direct exposure pressure, dependency spillover, retained leverage, residual role integrity, exposed versus retained core share, next-wave retained share and coherence, and a simple demand modifier from BLS growth.

Alongside that headline classifier, the live run now also computes a function-level transformation layer: function exposure pressure, retained function strength, retained accountability strength, retained bargaining power, delegation likelihood, and headcount displacement risk. Those function metrics do not replace the task graph; they sit above it and help explain whether exposed work is weakening the role's human-owned purpose or only changing how that purpose gets delivered.

In the current runtime, retained bargaining power is not just a simple average of task bargaining weights. It now leans more on pressure-adjusted retained task leverage, then blends in function-level bargaining retention and guardrails. That means routine or support-heavy work that is already under high pressure pulls the bargaining-power readout down instead of being over-credited by static task weights alone.

That matters because the model is trying to separate three different ideas that often get confused: tasks that are exposed, roles that get recomposed, and roles whose human-retained function actually weakens enough to collapse.

Demand modifier
$$M_{\text{demand}} = \operatorname{clamp}\!\left(\frac{\text{projectionGrowthPct}+2}{10},\,0,\,1\right)$$

High retained leverage, strong next-wave coherence, and low direct pressure push roles toward Reinforced or Narrowed upward. High demand growth can push some roles into Growing. Large exposed core share and weak retained integrity pull the model toward Shrinking, Polarized, or in the most severe cases Hollowed out. When the signals conflict, the model falls back to Unclear trajectory.

Recomposition layer

The page also shows a secondary recomposition panel. This is not the same as the final fate label. It is a narrower estimate of how much exposed work looks technically compressible and how much of that compression may convert into fewer labor hours in firms.

Compression and conversion
$$W = \operatorname{clamp}\!\left(\text{currentWaveAbsorbed}(1-0.20\cdot\text{couplingProtection})(1-0.10\cdot\text{functionRetention}),\,0,\,1\right)$$
$$O = \operatorname{clamp}\!\left(0.30\text{adoptionPressure} + 0.25(1-\text{couplingProtection}) + 0.20\text{currentWaveAbsorbed} + 0.10(1-f^{\text{accountability}}) + 0.08(1-f^{\text{judgment}}) + 0.07f^{\text{document}} - 0.05\text{signoff} - 0.05\text{liability},\,0,\,1\right)$$
$$\text{substitutionPotential} = W \cdot O, \qquad \text{substitutionGap} = W - \text{substitutionPotential}$$

Those raw values are then nudged by the task graph. More direct exposure pushes compression higher, while stronger retained leverage and dependency bottlenecks resist full organizational conversion. The current runtime also adds a structural routine-context lift here: occupations whose adaptation layer indicates high routine share, low people intensity, and lower job-zone complexity push compression upward more strongly in execution, workflow-admin, documentation, and some drafting-heavy bundles. That is why the recomposition panel can disagree with the final fate label: a role may have technically compressible work and still retain a strong human core.

Why the page looks the way it does

The visible role fate map is built from the underlying task table. Each column answers a different user question:

  • Current role: what takes up the role today?
  • Bargaining power: what most explains why this role exists?
  • Direct pressure: what AI pressures first?
  • Indirect spillover: what shrinks because it depends on exposed work?
  • Retained leverage: what still justifies the role after AI gets better?

Formally, the page ranks tasks by different task-level signals:

Client rankings
$$\text{Current role} = \operatorname{TopK}_t(s_t)$$
$$\text{Bargaining power} = \operatorname{TopK}_t\left(s_t \cdot \max(B_t,V_t)\right)$$
$$\text{Direct pressure} = \operatorname{TopK}_t(s_t \cdot p_t^{\text{direct}})$$
$$\text{Indirect spillover} = \operatorname{TopK}_t(s_t \cdot p_t^{\text{spill}})$$
$$\text{Retained leverage} = \operatorname{TopK}_t(\text{retainedShare}_t \cdot L_t)$$

What this model does not do

This model is useful, but it is not magic. It emits descriptive confidence and coverage fields, not a full probability distribution over the future.

  • Role-fate confidence is a margin-style score from the current rule set, not a calibrated probability.
  • Task evidence coverage still varies by occupation and task family, even after the reviewed evidence expansion.
  • User task input is now source-bucketed and function-aware, but it is still list-based rather than weighted by rough percentage buckets.
  • Multi-function roles are modeled better than before, but not every occupation yet has a fully reviewed multi-anchor function graph.
  • The classifier is calibrated heuristically against anchor occupations, not trained on a labeled dataset.
  • The wave system still exists internally, so role fate currently sits on top of the wave engine rather than fully replacing it.
